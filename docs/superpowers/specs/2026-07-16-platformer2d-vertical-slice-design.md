# 《尘白禁区》同人 2D 平台游戏：首个可玩切片设计

日期：2026-07-16  
状态：已完成分节确认，等待书面总审阅

## 1. 目标

从现有粗糙 Unity Demo 之外新建一个干净的 Unity 6.3 LTS 项目，制作一个 2～3 分钟、可完整通关的 2D 横版平台跳跃切片。切片首先验证角色移动、长关卡相机、关卡组织和资源管线，同时为多关卡、战斗以及 Live2D 对话剧情保留清晰扩展点。

项目首先面向 Windows/Steam 形态的桌面体验，输入和界面结构兼顾未来 Android/iOS 迁移。当前旧 Demo 仅作为行为参考，不直接迁移其场景和耦合脚本。

## 2. 项目性质与素材边界

本项目是以《尘白禁区》为题材的非官方、非商业同人创作，用于个人学习和同好分享。项目内应提供醒目的非官方同人声明，不进行销售、广告、付费解锁或捐赠变现。

允许参考官方角色设定和公开壁纸确定人物、世界观与视觉语言，但不把官网壁纸、游戏解包资源或他人二次创作直接打包进运行时。角色、背景和平台均重新绘制或生成并经过 2D 游戏适配；原始参考、生成提示词、生成日期和处理记录保存在 `ArtSource/`，可发布资源只进入 `Assets/Game/Art/`。

当前生成的场景基准图和芬妮侧视母版属于方向确认稿，不直接视为可绑定或可切片的最终运行时资产。

## 3. 首个切片范围

### 3.1 必须实现

- 2～3 分钟的单个原型关卡，主体横向推进，包含一小段纵向移动。
- 键盘水平移动和跳跃；Input System 中同时保留游戏手柄绑定结构。
- 精确、响应迅速的平台移动：土狼时间、跳跃输入缓冲、可变跳跃高度和空中控制。
- 稳定的地面检测、单向平台、坠落死亡与检查点复活。
- 3 个“异常晶体样本”收集物、可见收集反馈、计数显示和关卡终点。
- 能随角色左右、上下移动的 Cinemachine 相机，具有 Dead Zone、阻尼、前视和关卡边界限制。
- 灰盒地形、最小 UI、基础音效接口，以及可逐步替换的角色和环境表现层。

### 3.2 明确延期

- 普攻、技能、闪避、受击、敌人和战斗机关。
- 移动平台、升降机、斜坡、冰面滑行、可破坏墙体和动态天气。
- 多关卡菜单、存档、Addressables 远程内容和运行时区块流送。
- Live2D 角色、对话框、剧情编排和 Timeline 演出。
- 移动端触控 UI、商店 SDK、联网与任何商业化系统。

延期功能只能预留边界和数据入口，首个切片不得为了未来功能提前实现完整框架。

## 4. 技术基线

- 引擎：Unity 6.3 LTS。
- 渲染：Universal Render Pipeline，使用 2D Renderer。
- 核心包：Input System、Cinemachine 3、2D Animation、2D Sprite、2D Tilemap Editor、Unity Test Framework。
- Tilemap 自动拼接功能在创建项目时确认兼容的官方 Tilemap Extras 版本；若不兼容，首版使用普通 Tile Palette，不引入非必要替代插件。
- 非像素美术，不启用像素完美相机作为项目约束。
- 首个切片使用直接序列化引用；关卡目录接口与资源路径保持可迁移到 Addressables，但当前不引入 Addressables 复杂度。
- 不使用 ECS/DOTS、重型依赖注入容器、全局可写单例、按对象名称查找或把所有逻辑放入一个 `GameManager`。

## 6. 场景与模块架构

### 6.1 场景

- `00_Bootstrap`：初始化跨场景服务、输入、会话状态和场景切换入口。
- `10_Level_Prototype`：关卡区块、玩家出生点、碰撞 Tilemap、相机边界和关卡目标。
- `20_UI_Gameplay`：收集计数、淡入淡出和调试状态；首版可由 Bootstrap 叠加加载。

玩家预制体属于关卡运行上下文，由 Bootstrap 的组合根创建或注入依赖。切换关卡时不依赖场景对象名称，不保留重复玩家、重复输入系统或重复相机。

### 6.2 程序模块

- `Game.Core`：生命周期、通用事件、时间与场景抽象。
- `Game.Player`：输入读取、移动状态、物理运动和角色表现适配器。
- `Game.Level`：关卡上下文、区块、检查点、复活、终点和相机边界。
- `Game.Collectibles`：收集物、会话计数与收集事件。
- `Game.Presentation`：动画、相机、UI、音频和 VFX 响应。
- `Game.Infrastructure`：Unity 场景加载、存储占位和平台适配。
- 对应 EditMode 与 PlayMode 测试程序集。

模块通过小型接口和 C# 事件通信；可调参数保存在 ScriptableObject 配置中。跨场景服务由 Bootstrap 显式持有并注入，不建立任意对象都能修改的全局状态。

## 7. 玩家控制设计

`PlayerInputReader` 只把 Input Actions 转换为方向、按下、保持和释放信号。首版 `Move` 绑定 A/D、左右方向键、手柄左摇杆与十字键，`Jump` 绑定空格和手柄南侧按钮。`PlayerMotor2D` 负责速度和 Rigidbody2D 写入；`PlayerStateMachine` 管理 Grounded、Rising、Falling、Landing 等状态；`PlayerPresenter` 将状态转换为动画与表现事件。

数据流如下：

```text
Input Actions
  -> PlayerInputReader
  -> PlayerStateMachine / PlayerMotor2D
  -> PlayerStateSnapshot
  -> PlayerPresenter + CameraTarget + VFX/Audio
```

土狼时间、跳跃缓冲、最大速度、加速度、减速度、空中控制、上升重力和下落重力均为配置数据。地面检测使用 LayerMask 和明确的 2D Cast，不依赖 Tag、对象名或摩擦力碰巧阻止滑动。角色碰撞体与美术轮廓分离，头发、裙摆和武器不参与主体碰撞。

移动碰撞与动画完全解耦：灰盒胶囊或盒形碰撞体先通过手感验收，之后更换芬妮贴图不得改变跳跃距离和落地判定。

## 8. 相机与长关卡基础

Cinemachine 相机跟随独立的 `CameraTarget`，不直接跟随带有呼吸或落地动画的角色 Sprite 根节点。

- 水平方向使用 Dead Zone、轻度阻尼和面向方向前视。
- 垂直方向使用更克制的阻尼，避免小跳时画面频繁上下抖动。
- 每个关卡或区块提供 Confiner 2D 边界；进入新区块时由关卡上下文切换边界，而不是由关卡脚本直接改写相机坐标。
- 坠落期间相机不会无限跟随角色越出关卡范围。
- 相机参数存入独立配置，允许后续为探索、战斗和剧情创建不同 Profile。

Dead Zone 表示玩家可在屏幕中心区域内移动而不立即推动相机。它用于消除细小运动造成的晃动，不是玩家不可进入或看不见的区域。

## 9. 关卡组织与环境资源

采用 Tilemap 与 Prefab 混合方案。

### 9.1 Tilemap 分工

- `Gameplay Tilemap`：唯一的主要静态地形碰撞来源，使用 Tilemap Collider 2D 与 Composite Collider 2D。
- `Terrain Art Tilemap`：积雪表面、金属边缘、断面、墙面和雪层覆盖，不决定玩法几何。
- 逻辑网格和玩家尺度在灰盒阶段确定，贴图 PPU 之后按网格适配，不能反向用图片尺寸决定跳跃参数。

首批 Tile 包含表面、内部填充、左右边缘、内外角、竖直墙面、底部以及完整、轻破损、重破损变化。首版不包含斜坡和动态 Tile。

### 9.2 Prefab 分工

- 短、中、长三种静态单向平台。
- 视觉支架和悬空底座。
- 出生／复活信标与关卡终点装置。
- 护栏、路灯、车辆、隔离路障、设备箱、管线、雪堆、碎石和三档异常晶体。

装饰默认无碰撞；需要影响玩法的物件必须使用明确的 Gameplay 组件和独立碰撞层，不能通过美术 Sprite 自动产生碰撞。

### 9.3 Level Chunk

每个 `Level Chunk` 包含碰撞 Tilemap、美术 Tilemap、装饰根节点、视差锚点、出生点、相机边界与验证组件。首个切片可以只使用一个场景和若干区块根节点，但结构允许以后把区块拆为追加加载场景。

## 10. 美术方向与背景

首关是“零区冰封城市／异常晶体禁区”，采用阴云下的冷色白昼。环境保持可信尺度，但以动漫化方式简化材质和细节，不做玩具化全 Q 版，也不使用高密度照片写实。

- 主色：低饱和白、冷灰、蓝灰和淡青。
- 发光：异常晶体使用克制的青蓝发光，避免满屏赛博霓虹。
- 焦点：芬妮的金发、橙红服装和少量荧光绿点缀构成画面视觉焦点。
- 表面：积雪混凝土、磨损涂装钢材、霜玻璃和少量晶体半透明感。
- 可读性：平台边缘和危险区域优先于环境细节，禁止用高频装饰破坏角色轮廓。

背景分为独立视差层：

- `Background_Sky`：阴云和远雾，可横向延展。
- `Background_Far`：城市天际线和巨型封锁设施。
- `Background_Mid`：建筑、道路残骸和隔离设施。
- `Background_Near`：近景钢架、管线和局部遮挡物。
- 独立雾层用于弱化远景和遮盖循环接缝。

## 11. 芬妮角色与动画资源

芬妮·戈尔登采用非像素、约 4 头身的动漫 Q 版比例。角色保留以下识别锚点：金色卷发双马尾、黑色机械发饰、橙红／黑白服装、荧光绿点缀、机械护臂、左右腿不对称设计和战术短靴。

正常跑跳时武器收纳在背部，攻击或瞄准时才取出。骨骼预留背部挂点和双手持枪点，但首版不制作攻击动画。

### 11.1 动画技术

采用混合方案：常规循环动作使用 Unity 2D Animation 骨骼；起跳、落地、急停以及未来闪避、受击等强调姿势可以使用 Sprite Swap 或替换关键帧；特效使用逐帧动画。

角色最终拆为约 20～24 个可独立控制部件，包括头部、前后发、卷发段、身体、骨盆、裙片、四肢、机械护臂、枪套和背部武器。主体约 18～22 根骨骼，卷发和裙片增加少量辅助骨骼。最终拆件必须保留关节遮挡余量并人工整理边缘，不能把生成整图自动切割后直接绑定。

### 11.2 首批动画

- Idle
- Run
- Jump Start
- Air Rise
- Air Fall
- Land
- Brake / Turn

脸、手型和武器建立 Sprite Library 分类，便于后续替换。只制作右向母资源，运行时翻转角色根节点显示左向；非对称服装的镜像属于当前玩法表现约定，若未来剧情或特写要求真实左右关系，再制作专用方向资源。

## 12. 收集、复活与关卡完成数据流

收集物触发后只发布收集事件，由会话状态更新计数，再由 UI 和 VFX 分别响应。收集物本身不直接查找或修改 UI。

```text
Collectible Trigger
  -> CollectibleCollected event
  -> LevelSessionState
  -> HUD counter + collect VFX/audio
```

检查点激活后向 `RespawnService` 注册安全位置。坠落 Trigger 请求角色复活；若当前检查点无效，则回退到关卡默认出生点。终点只在玩家进入时发布完成事件，不直接加载下一个场景；场景流程由关卡会话或未来的关卡导演处理。

## 13. 错误处理与编辑器校验

- 必需的序列化引用在 `OnValidate` 和启动阶段校验；开发构建中给出带对象路径的明确错误。
- Bootstrap 缺少核心配置时停止进入玩法场景，而不是让系统以半初始化状态运行。
- 相机边界缺失时允许使用明确记录的无边界降级模式，但必须产生一次警告。
- 检查点无效时回退到关卡默认出生点；默认出生点也缺失则停止复活并报告配置错误，避免无限循环。
- Level Chunk 提供编辑器校验，检查重复 ID、缺失边界、错误 Layer、装饰碰撞和越界出生点。
- 事件订阅在组件停用或销毁时成对解除，防止换场景后的重复响应。

## 14. 测试与验收

### 14.1 EditMode

- 土狼时间、跳跃缓冲和可变跳跃窗口的纯逻辑测试。
- 玩家状态转换和重复输入边界测试。
- 收集计数、重复收集保护和关卡完成条件测试。
- Level Chunk、配置引用和唯一 ID 验证测试。

### 14.2 PlayMode

- 玩家能稳定站立、跨越平台接缝、从单向平台正确落下和落地。
- 不同渲染帧率下移动与跳跃结果处于允许误差内。
- 坠落后回到最近检查点，且速度、状态和相机正确复位。
- 收集 3 个样本后 HUD 正确，重新进入已收集 Trigger 不重复计数。
- 相机 Dead Zone、前视、纵向跟随与 Confiner 在关卡边缘无明显抖动或穿帮。
- 重载原型关卡不会产生重复输入、重复玩家、重复相机或事件订阅。

### 14.3 切片完成标准

- 从启动场景进入关卡后无需编辑器干预即可完成一次完整流程。
- 关卡时长约 2～3 分钟，包含横向推进和一段清晰纵向变化。
- 移动手感、碰撞和相机在灰盒状态即通过验收，美术替换不改变玩法参数。
- 所有运行时资源位于规定目录，控制台无持续错误或重复警告。
- Windows 开发构建可启动、游玩、收集、坠落复活并到达终点。

## 15. 性能与后续扩展边界

首版避免过早优化，但建立可测量基线：目标桌面环境稳定 60 FPS；Tilemap 静态碰撞合并；重复小图进入 Sprite Atlas；限制近景半透明和大面积晶体发光造成的 Overdraw；角色 Sprite Skin 和粒子数量保持可分析。

未来扩展遵循以下入口：

- 战斗：在玩家状态机外增加独立能力／战斗模块，复用持枪挂点和输入 Action Map。
- 多关卡：以 Level Catalog 和场景流程服务管理，不让关卡自己硬编码下一场景名称。
- 长关卡：Level Chunk 可升级为追加加载场景，并在出现实际资源压力后引入 Addressables。
- Live2D 剧情：作为独立 Presentation 和 Narrative 模块叠加，不与游戏内骨骼 Sprite 管线混用。
- 移动端：复用 Input Actions 和游戏逻辑，增加触控适配与平台资源配置，不复制一套玩家控制器。

## 16. 新项目创建前的最终结论

设计允许立即进入实施计划。新项目创建时先搭建目录、程序集、Bootstrap、输入、灰盒玩家、相机和原型关卡；角色拆件、最终 Tile、背景层和特效随后按已定义的资源接口逐步替换。新建项目不以“所有贴图已完成”为前置条件。
